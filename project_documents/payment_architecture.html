Payment Architecture Documentation

A complete guide to understanding, modifying, and managing the Stripe & SSLCommerz payment integration.

1. How the Payment Flow Works (Webhook-First)

This project uses a highly reliable Server-to-Server (Webhook-First) architecture. This means we do not rely on the user's browser returning to our success page to finalize the order. Instead, the payment gateways communicate directly with our server in the background.

  1. Checkout: User clicks "Pay". System creates an order in PENDING/UNPAID state and redirects to Stripe/SSLCommerz.
  2. Payment: User completes the payment on the gateway's secure page.
  3. Background Webhook (Crucial Step):
    • Stripe silently sends a POST request to /payment/stripe/webhook.
    • SSLCommerz silently sends an IPN to /payment/ssl/ipn.
    • The server verifies this signal, updates the database, deducts stock, and sends a Success Email.
  4. User Redirection (Frontend UI): Meanwhile, the user is redirected back to our website's /success page. Because the background webhook might take a few seconds, the UI shows a "Processing Your Payment" spinner.
  5. AJAX Polling: The UI pings our server every 3 seconds to check if the background webhook finished. Once confirmed, it redirects the user to their final Thank You screen.

2. Code Locations (Where is what?)

Component File Path Responsibility
Routes routes/web.php Defines the endpoints (success, cancel, webhook, status) for gateways.
Stripe Controller app/Http/Controllers/Payment/StripeController.php Handles Stripe Webhook signature verification and AJAX status checking.
SSL Controller app/Http/Controllers/Payment/SSLCommerzController.php Handles SSL IPN (Instant Payment Notification) verification.
Database Logic app/Repositories/OrderRepository.php The finalizeOrder() method handles stock deduction, DB updates, and triggers the Email.
Processing UI resources/views/web/payment/processing.blade.php The "Loading" screen that polls the server via Javascript.
Success Email app/Mail/OrderConfirmedMail.php
resources/views/emails/order_confirmed.blade.php		 The backend class and the HTML template for the email sent to the user.
CSRF Exception bootstrap/app.php Ensures Laravel does not block the POST webhooks due to missing CSRF tokens.

3. How to Edit or Update Features

Modifying the Success Email Design

To change how the email looks, open resources/views/emails/order_confirmed.blade.php. It is a standard HTML file with inline CSS (required for emails) and Blade variables (like $order->order_number).

Adding a New Payment Gateway

If you add a new gateway (e.g., PayPal), follow the Webhook-First rule:

  • The user should be redirected to the gateway via OrderRepository.
  • Create a Webhook/IPN endpoint that verifies the payment in the background and calls $this->orderRepository->finalizeOrder().
  • The user's success redirect should simply return view('web.payment.processing').

Modifying Post-Payment Database Logic

If you need to give the user points, deduct wallet balance, or notify an admin after a successful payment, add your logic inside app/Repositories/OrderRepository.php inside the finalizeOrder() method.

4. API Keys & Setup Instructions

Stripe Setup

1. Get Public & Secret Keys

  • Go to Stripe Dashboard -> Developers -> API Keys.
  • Copy the Publishable key and Secret key into your .env file.

2. Get Webhook Secret

  • Go to Stripe Dashboard -> Developers -> Webhooks.
  • Click "Add endpoint".
  • URL: https://yourdomain.com/payment/stripe/webhook
  • Events: Select checkout.session.completed.
  • Click "Add endpoint", then click "Reveal" next to Signing secret.
  • Copy the whsec_... key into your .env file.
STRIPE_KEY=pk_test_... STRIPE_SECRET=sk_test_... STRIPE_WEBHOOK_SECRET=whsec_...

SSLCommerz Setup

  • You get the Store ID and Store Password via email when you register on SSLCommerz.
  • For Sandbox (Test Mode), set SSLCOMMERZ_IS_SANDBOX=true. For Live, set it to false.
SSLCOMMERZ_STORE_ID=your_store_id SSLCOMMERZ_STORE_PASSWORD=your_store_password SSLCOMMERZ_IS_SANDBOX=true
